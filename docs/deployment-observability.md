# Deployment observability

## Framework: deployments = tagged releases

WannaBuild has no production servers. "Deployment" here means **a tagged
release publishing the framework artifacts** (skills, plugins, scripts) for
consumers to install.

The artifacts that prove what shipped:

| Artifact | Source |
|---|---|
| `CHANGELOG.md` | Generated by release-please from Conventional Commits. |
| Release notes (GitHub) | Generated by release-please. |
| Git tags (`vX.Y.Z`) | Created by release-please. |
| Plugin/skill marketplace entries | Updated by maintainers via `scripts/install-claude-skill.sh` / `scripts/install-codex-skill.sh` install paths. |

### How to subscribe to releases

- **GitHub**: Watch → Custom → Releases on the repo.
- **RSS**: `https://github.com/<owner>/<repo>/releases.atom`.
- **Slack** (recommended): use the official `release-please` Slack action or
  the `slackapi/slack-github-action` step inside the release workflow to post
  release notes to a channel. Snippet:

  ```yaml
  - name: Notify Slack on release
    if: ${{ steps.release.outputs.release_created }}
    uses: slackapi/slack-github-action@v1
    with:
      payload: |
        {
          "text": ":rocket: WannaBuild ${{ steps.release.outputs.tag_name }} released",
          "blocks": [
            {"type":"section","text":{"type":"mrkdwn","text":"*${{ steps.release.outputs.tag_name }}* shipped.\nNotes: <${{ steps.release.outputs.html_url }}|GitHub release>"}}
          ]
        }
    env:
      SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
  ```

### "Who shipped what" runbook fragment

1. `git log --oneline <prev_tag>..<current_tag>` for the commit list.
2. The release-please PR for the version is the canonical change set.
3. CHANGELOG.md is the human-readable summary.
4. For binary/plugin install issues, the install scripts pin to the released
   version: `scripts/install-claude-skill.sh` and
   `scripts/install-codex-skill.sh`.

## Target-project guidance

For real applications, deployment observability has three pillars:

### 1. Deploy markers

Every deploy should annotate every relevant dashboard. A spike in errors
is meaningless without knowing what shipped.

- **Datadog**: `events.post` with tag `source:deploy` — Datadog dashboards
  show vertical lines automatically. See the
  [Events API](https://docs.datadoghq.com/api/latest/events/).
- **Grafana**: dashboard annotations via the
  [Annotations API](https://grafana.com/docs/grafana/latest/developers/http_api/annotations/).
- **New Relic**: deploy markers via the v2 API (POST to
  `/v2/applications/<id>/deployments.json`).
- **Sentry**: send a release before the deploy, then a deploy event.
- **Honeycomb**: marker events via the API.

### 2. Progressive rollout observability

When you canary or feature-flag a release:

- Tag every metric/log/trace with `release` and `rollout_phase` (`canary`,
  `25%`, `100%`).
- Build dashboards that split each top-level SLO by `release`. Compare
  canary vs baseline before scaling.
- For OpenTelemetry, set this as a resource attribute so every signal is
  tagged consistently:

  ```text
  service.version=2026.05.02-canary
  deployment.environment=production
  ```

- For feature flags, emit a counter per `(flag, variant)` so you can see
  which variant a user got when an error fired.

### 3. Correlate deploys with error spikes

Two patterns:

- **Sentry release health**: connects errors to releases automatically when
  you set `SENTRY_RELEASE`. The "Releases" view shows "crash-free sessions"
  delta vs prior release.
- **Datadog APM**: the "Deployments" view ties trace-level errors to deploys.
- **Grafana / Prometheus**: use `info`-style series with the release label,
  combined with annotations, to manually overlay release markers.

### 4. Recommended dashboards

Bookmark these for every service:

| Dashboard | Purpose |
|---|---|
| **Service overview** | RED metrics, SLO status, error budget burn. |
| **Release health** | Errors / sessions / latency split by release. |
| **Dependencies** | DB, cache, downstream HTTP — RED metrics each. |
| **Top errors** | Sentry-style top issues this week. |
| **Top slow endpoints** | p99 latency leaderboard. |
| **Cost** | If on a SaaS metrics backend, ingestion volume by service. |

### Runbook fragments

Cross-link from your alerts:

- "Deploy spiked errors" → revert via your deploy tool; gather Sentry release
  diff; open post-mortem.
- "Canary metrics regress" → halt rollout; collect comparison snapshot;
  decide rollback vs forward-fix.
